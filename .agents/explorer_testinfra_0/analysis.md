# Loyea 项目测试基础结构探测与 E2E 测试用例设计分析报告

本报告记录了对 Loyea 项目的测试基础设施探测结果，并针对核心需求（R1 MCP管理、R2 工具调用、R3 物理感知、R4 后台问候）设计了 50 个端到端 (E2E) 测试用例。

---

## 一、 测试基础结构探测结果

经过对项目根目录及 `app` 模块的深度扫描，探测到以下关于构建与测试配置的事实：

### 1. Gradle 配置文件与版本信息
*   **Gradle 版本**：项目采用 Gradle 8.13，分发包下载地址配置在 `gradle/wrapper/gradle-wrapper.properties` 中，使用了腾讯云镜像源（`https://mirrors.cloud.tencent.com/gradle/gradle-8.13-bin.zip`）。
*   **Android Gradle 插件 (AGP) 版本**：`8.13.2`，定义于根目录 `build.gradle.kts`。
*   **Kotlin 版本**：`1.9.22`。
*   **Java 版本配置**：编译与目标 Compatibility 均配置为 Java 17 (`sourceCompatibility = JavaVersion.VERSION_17`，`jvmTarget = "17"`）。
*   **本地 SDK 路径**：在 `local.properties` 中指定为 `D:\Dev\Android\Sdk`。

### 2. 测试依赖项与能力
在 `app/build.gradle.kts` 中，已声明了以下主流的 Android 测试依赖，具备了单元测试和 UI/E2E 仪器化测试的完备基础：
*   **本地单元测试依赖**：
    *   `testImplementation("junit:junit:4.13.2")`：用于运行 JVM 上的单元测试。
*   **Android 仪器化/E2E测试依赖**：
    *   `androidTestImplementation("androidx.test.ext:junit:1.1.5")`：AndroidX 测试运行器。
    *   `androidTestImplementation("androidx.test.espresso:espresso-core:3.5.1")`：用于传统 View 或底层输入模拟。
    *   `androidTestImplementation(platform("androidx.compose:compose-bom:2024.02.00"))`：Compose 依赖管理。
    *   `androidTestImplementation("androidx.compose.ui:ui-test-junit4")`：Compose UI 核心测试规则与匹配器。
    *   `debugImplementation("androidx.compose.ui:ui-test-manifest")`：在测试期间加载 Activity 容器。

### 3. 现存测试类分析
*   **目录扫描**：经探测，`app/src/` 目录下**仅存在** `main` 源码集，**完全不存在** `test`（本地单元测试）和 `androidTest`（仪器化测试）这两个标准的 Android 测试目录。
*   **结论**：Loyea 项目目前是一个“零测试”状态的项目，没有任何现存的测试类。后续所有的测试代码都需要从头建立相应的测试套件目录。

### 4. Windows PowerShell 环境下运行构建与测试命令
在 Windows PowerShell 环境中，必须以 `.\` 作为当前目录下可执行脚本的前缀。以下为常用命令：
*   **编译构建 Debug 产物**：
    ```powershell
    .\gradlew assembleDebug
    ```
*   **运行本地单元测试**：
    ```powershell
    .\gradlew test
    # 或者针对特定的 app 模块：
    .\gradlew :app:testDebugUnitTest
    ```
*   **运行 E2E/Android 仪器化测试**（需连接真机或启动 Android 模拟器）：
    ```powershell
    .\gradlew connectedAndroidTest
    # 或者针对特定的 app 模块：
    .\gradlew :app:connectedDebugAndroidTest
    ```

## 二、 E2E 测试用例设计 (共 50 个)

根据对 R1-R4 功能特性的分析，独立设计以下 50 个测试用例，划分为 4 个 Tier。

---

### Tier 1: 功能覆盖测试 (Feature Coverage) - 共 20 个 (每项需求 5 个)

#### 【R1. MCP 客户端协议与多服务器管理】
1.  **用例 R1-T1-01：添加 Mock MCP 服务端**
    *   **测试目的**：验证用户在设置界面中能正确添加并启用一个新的 Mock MCP 服务器配置。
    *   **测试步骤**：
        1. 进入“设置” -> “API配置”。
        2. 点击“添加MCP服务器”，输入名称 "Mock_MCP"、连接方式 "SSE" 和 URL "http://10.0.2.2:8000/sse"。
        3. 点击“保存”。
    *   **预期结果**：设置列表中成功展示 "Mock_MCP" 卡片，SharedPreferences 中持久化写入该配置。
2.  **用例 R1-T1-02：展示 MCP 服务器工具 (Tools) 列表**
    *   **测试目的**：验证成功连接 MCP 服务器后，客户端能拉取并渲染其暴露的 Tools。
    *   **测试步骤**：
        1. 启动已配置 "Mock_MCP" 的 Loyea 客户端。
        2. 在 API 配置面板中，点击展开 "Mock_MCP" 卡片。
    *   **预期结果**：卡片内部展示该服务器暴露的所有 Tools 的名称及描述（例如 `calculator`，`weather_fetcher`）。
3.  **用例 R1-T1-03：编辑并更新 MCP 服务端配置**
    *   **测试目的**：验证编辑 MCP 服务器配置能实时应用并正确持久化。
    *   **测试步骤**：
        1. 在 MCP 列表中点击 "Mock_MCP" 的编辑图标。
        2. 修改 URL 为 "http://10.0.2.2:9000/sse"，保存。
    *   **预期结果**：配置信息成功更新，重新读取 SharedPreferences 确认 URL 已变为新的地址。
4.  **用例 R1-T1-04：实时连接状态检测（在线与断开）**
    *   **测试目的**：验证 UI 面板可正确流转显示已连接、连接中和已断开状态。
    *   **测试步骤**：
        1. 开启一个 Mock MCP 本地服务，Loyea 客户端与其连接，观察 UI 状态。
        2. 关闭该 Mock 本地服务，观察 UI 状态。
    *   **预期结果**：服务开启时卡片显示“已连接”（带有绿色标识）；服务关闭后 UI 迅速转为“已断开”（带有灰色或红色警告标识）。
5.  **用例 R1-T1-05：删除 MCP 服务端**
    *   **测试目的**：验证删除 MCP 服务器后，其配置及状态被完全移除。
    *   **测试步骤**：
        1. 在设置的 MCP 列表中点击 "Mock_MCP" 的删除按钮，确认删除。
    *   **预期结果**：UI 列表不再包含该卡片，SharedPreferences 内该服务器的 JSON 数据被彻底清空。

#### 【R2. LLM 工具调用链路闭环】
6.  **用例 R2-T1-01：工具定义转换与注入大模型请求**
    *   **测试目的**：验证当前启用的所有 MCP Tools 能被包装为符合 OpenAI/DeepSeek 标准的格式注入到大模型请求的 `tools` 参数中。
    *   **测试步骤**：
        1. 启用 MCP 服务器 "Mock_MCP"（包含 `calculator` 工具）。
        2. 在主聊天界面输入“帮我计算一下 123 + 456”。
        3. 拦截向大模型发出的 HTTP Payload。
    *   **预期结果**：Payload 的 `tools` 字段中包含了 `calculator` 工具的详细 JSON 模式定义。
7.  **用例 R2-T1-02：大模型 Tool Call 响应拦截**
    *   **测试目的**：验证客户端能正确识别并拦截 LLM 返回的调用工具指示。
    *   **测试步骤**：
        1. 发送需要调用工具的消息，大模型返回带有 `tool_calls` 的 SSE 数据流或非流式响应。
        2. 监控客户端的响应流拦截解析器。
    *   **预期结果**：客户端成功解析出 `tool_calls` 对象，包含具体的 ID、工具名 `calculator` 和参数 `{"expression": "123+456"}`。
8.  **用例 R2-T1-03：JSON-RPC 工具调用请求与响应**
    *   **测试目的**：验证拦截 Tool Call 后，客户端可通过 JSON-RPC over HTTP/SSE 向目标服务器执行请求并获取输出。
    *   **测试步骤**：
        1. 拦截到 `tool_calls` 请求后，客户端向 `http://10.0.2.2:8000/sse` 发送 JSON-RPC 调用。
        2. 拦截客户端接收的工具执行回包。
    *   **预期结果**：JSON-RPC 发送成功，服务器返回执行结果 `{"result": "579"}`。
9.  **用例 R2-T1-04：Tool 角色消息追加与下一轮对话触发**
    *   **测试目的**：验证客户端将执行结果作为 `tool` 消息回填至历史，并自动发起下一轮对话。
    *   **测试步骤**：
        1. 获取工具执行结果 "579"。
        2. 观察客户端重新向大模型发出的下一轮请求。
    *   **预期结果**：请求的 `messages` 队列中新增了 `role: tool`、`content: 579` 以及之前对应的 `assistant` 角色 Tool Call 消息，且成功收到大模型的最终自然语言回答。
10. **用例 R2-T1-05：McpCallItem 状态流转与 UI 渲染**
    *   **测试目的**：验证聊天气泡中 `McpCallItem` 组件在运行、成功或失败时的 UI 呈现。
    *   **测试步骤**：
        1. 发起调用工具对话，观察聊天列表。
        2. 在调用执行期间，查看该工具卡片图标；执行完毕后再次查看。
    *   **预期结果**：执行中卡片显示为 `RUNNING` 且齿轮图标保持旋转；执行完毕后卡片状态立即转为 `SUCCESS`，图标变为绿色对勾。

#### 【R3. 智能手表与定位物理感知集成】
11. **用例 R3-T1-01：系统时间参数注入**
    *   **测试目的**：验证开启物理感知后，系统当前时间被准确注入系统 Prompt。
    *   **测试步骤**：
        1. 开启“使用系统时间”及“物理感知”选项。
        2. 在聊天中发送任意消息，拦截系统 Prompt。
    *   **预期结果**：系统 Prompt 头部含有结构化字段 `[Physical Context]`，其中包括当前的年月日、时分秒以及星期信息。
12. **用例 R3-T1-02：定位获取及无权限降级**
    *   **测试目的**：验证拒绝定位权限后，系统能无缝降级到模拟位置。
    *   **测试步骤**：
        1. 进入设置开启物理感知定位，在系统弹窗中选择“拒绝定位权限”。
        2. 发送消息，查看注入的坐标参数。
    *   **预期结果**：定位数据自动回退至预设的北京模拟坐标 `39.9042, 116.4074`。
13. **用例 R3-T1-03：虚拟心率生成器波动生成**
    *   **测试目的**：验证“虚拟手表心率生成器”在开启物理感知时，能持续产生合理范围内的波动数值。
    *   **测试步骤**：
        1. 在设置中开启“虚拟手表心率生成器”，心率模拟状态保持默认。
        2. 监控心率数据产出。
    *   **预期结果**：生成的模拟心率数据在 `60-100 bpm` 之间随机小幅波动，没有任何 NaN 或越界值。
14. **用例 R3-T1-04：静息与运动模拟状态切换**
    *   **测试目的**：验证切换“静息/运动”状态后，注入的心率数据分类和数值随之更新。
    *   **测试步骤**：
        1. 在感知设置中将模拟状态由“静息”切换为“运动”。
        2. 发送一条消息，检查发送 payload 中的 Prompt 内容。
    *   **预期结果**：Prompt 中的 Heart Rate 变为代表运动状态的数值范围（如高于 100 bpm），并且状态标识更新为 (Active/Sport)。
15. **用例 R3-T1-05：智能手表同步开关 UI 与 mock 接口预留**
    *   **测试目的**：验证“手表同步”硬件开关 UI 的交互以及底层接口的 Mock 注入。
    *   **测试步骤**：
        1. 进入物理感知设置页，找到并切换“手表同步”开关。
        2. 检查底层 `WatchDataRepository` 的 Mock 代码是否可被测试框架控制注入。
    *   **预期结果**：UI 交互正常响应，Mock 接口能稳定向上层提供模拟的手表传感器数据。

#### 【R4. 后台定时主动问候推送】
16. **用例 R4-T1-01：WorkManager 定时任务触发**
    *   **测试目的**：验证后台定时任务能被 WorkManager 正常注册和拉起。
    *   **测试步骤**：
        1. 启动 App 并使得后台任务成功注册。
        2. 使用 Android 测试框架或 adb 命令模拟定时时间已到，强行触发 WorkManager 任务。
    *   **预期结果**：后台问候任务被成功唤醒，开始执行后台逻辑。
17. **用例 R4-T1-02：后台静默大模型请求发起**
    *   **测试目的**：验证后台任务唤醒后，能在无 UI 的情况下静默请求 LLM 并注入感知上下文。
    *   **测试步骤**：
        1. 触发后台定时任务，监听网络请求。
    *   **预期结果**：后台发起了一个向大模型接口的 Chat Completion 请求，Prompt 中成功融入了当前时间的深夜属性及模拟心率等物理参数。
18. **用例 R4-T1-03：后台问候内容生成与状态栏通知展示**
    *   **测试目的**：验证大模型生成的问候能够触发格式优雅的系统 Notification。
    *   **测试步骤**：
        1. 大模型返回生成的 50 字以内关怀话语。
        2. 检查系统通知栏。
    *   **预期结果**：状态栏弹出新通知，通知标题为当前激活的角色姓名，内容为大模型生成的关怀语。
19. **用例 R4-T1-04：点击通知拉起 MainActivity**
    *   **测试目的**：验证点击状态栏通知后，App 被引导启动并跳转到对应的聊天页面。
    *   **测试步骤**：
        1. 在手机通知栏点击生成的 Loyea 主动问候通知。
    *   **预期结果**：Loyea 客户端被推前台，`MainActivity` 被拉起，并处于当前聊天会话页。
20. **用例 R4-T1-05：后台问候消息持久化与会话历史插入**
    *   **测试目的**：验证拉起界面后，该条主动问候消息已持久化并插入到聊天对话流中。
    *   **测试步骤**：
        1. 在点击通知拉起 App 后，检查当前聊天页的历史气泡列表。
        2. 重启 App，检查会话历史。
    *   **预期结果**：聊天列表中在最底部显示刚才的 AI 主动问候，且该条消息已成功保存至本地 JSON 缓存文件。

### Tier 2: 边界与异常测试 (Boundary & Corner Cases) - 共 20 个 (每项需求 5 个)

#### 【R1. MCP 客户端协议与多服务器管理】
21. **用例 R1-T2-01：无效的 MCP URL 格式验证**
    *   **测试目的**：验证输入非法格式 URL（如缺少协议头 `10.0.2.2` 或包含特殊控制字符）时，UI 能拦截并给出明确提示。
    *   **预期结果**：输入框变红或弹出 Toast 提示“URL 格式不合法”，阻止点击保存。
22. **用例 R1-T2-02：空白 MCP 配置状态**
    *   **测试目的**：验证当无任何 MCP 服务器配置时，系统核心对话模块及配置页面均能优雅降级展示。
    *   **预期结果**：聊天界面不报错，API 配置列表显示“暂无配置任何 MCP 服务端”，且 SharedPreferences 没有异常脏数据。
23. **用例 R1-T2-03：相同名字的工具冲突处理**
    *   **测试目的**：验证两个 MCP 服务端提供同名 Tool（例如都提供 `get_weather`）时，客户端注册与调用的路由隔离性。
    *   **预期结果**：客户端可以通过在 Tool 声明中加入服务前缀（如 `ServerA__get_weather` 和 `ServerB__get_weather`）避免模型调用混乱，或合理地只调用已勾选/激活的主服务器。
24. **用例 R1-T2-04：低网速/高延迟下的 SSE 连接超时**
    *   **测试目的**：模拟极端慢速网络环境下建立 MCP SSE 连接。
    *   **预期结果**：连接状态显示“连接中”超过阈值（如 10s）后，自动转换为“已断开”并伴随友好错误信息，避免永久阻塞 UI。
25. **用例 R1-T2-05：SharedPreferences 中 MCP 数据文件损坏**
    *   **测试目的**：SharedPreferences 持久化文件遭到破坏（篡改为非法 JSON），App 启动时的防崩溃与降级恢复。
    *   **预期结果**：App 冷启动正常，不闪退，自动将受损的数据清空重置为默认空白或预置 MCP 配置。

#### 【R2. LLM 工具调用链路闭环】
26. **用例 R2-T2-01：大模型并行 Tool Calls 拦截与顺序/并发执行**
    *   **测试目的**：验证 LLM 在一个回答中同时给出多个 `tool_calls` 时，客户端的处理能力。
    *   **预期结果**：客户端在 UI 渲染两个并行的 `McpCallItem` 卡片，分别向对应的服务器发起 JSON-RPC，收集所有执行结果后，一次性以多个 `tool` 角色消息提交回 LLM，保证链路不乱。
27. **用例 R2-T2-02：MCP 服务端异常回包/500错误容错**
    *   **测试目的**：测试当 MCP 服务端执行出错，或者返回完全不是 JSON 格式的数据时，客户端的容错能力。
    *   **预期结果**：`McpCallItem` 卡片标记为 `FAILED`，将原始的错误文本作为 `tool` 消息内容发给 LLM，大模型根据错误信息生成友好的引导式降级回复（例如“抱歉，计算工具暂时无法工作...”）。
28. **用例 R2-T2-03：深度 Tool Calls 循环递归限制**
    *   **测试目的**：防止大模型陷入工具死循环调用（例如：工具A输出促使模型调用工具B，工具B输出又促使模型调用工具A，无限循环）。
    *   **预期结果**：当检测到同一轮对话中的累计 Tool Call 调用深度达到阈值（如 5 次）时，客户端强制打断并终止调用链，将状态设为 `FAILED`，大模型直接生成文本回答。
29. **用例 R2-T2-04：工具卡片展开折叠时状态突变 UI 防闪烁**
    *   **测试目的**：验证工具在 RUNNING 期间用户点击展开，然后状态突变到 SUCCESS 时，UI 的平滑性。
    *   **预期结果**：展开高度跟随内容自适应平滑收缩或展开，齿轮旋转到绿勾切换的过渡动画自然，不会发生卡顿或文字瞬间闪烁。
30. **用例 R2-T2-05：发送中二次消息发送阻断**
    *   **测试目的**：在 Tool Call 执行挂起（UI 呈现 `isThinking` 和 `RUNNING` 状态）时，阻止用户再次在输入框发送新内容。
    *   **预期结果**：聊天输入框和发送按钮被正确置灰（禁用），防止产生竞争冲突与状态污染。

#### 【R3. 智能手表与定位物理感知集成】
31. **用例 R3-T2-01：用户输入非法模拟坐标校验**
    *   **测试目的**：验证在设置中手动输入模拟经纬度坐标时的边界值检查。
    *   **预期结果**：输入例如 `Latitude: 100, Longitude: 200` 等超出地理边界（纬度±90，经度±180）的值时，提示“坐标范围无效”，保存按钮禁用。
32. **用例 R3-T2-02：运行时位置权限中途关闭的防抖降级**
    *   **测试目的**：用户在 App 运行对话过程中，切回系统设置强行关闭定位权限，再切回 App 发送消息。
    *   **预期结果**：App 不会抛出 `SecurityException` 崩溃，而是通过 try-catch 拦截，并在本次消息发送中静默回退至模拟定位坐标注入。
33. **用例 R3-T2-03：手表同步开启但硬件掉线降级**
    *   **测试目的**：开启了“手表同步”，但是 WearOS 蓝牙断开或 Health Connect API 拒绝服务。
    *   **预期结果**：心率数据无缝回退到内置的虚拟心率波动模拟器，维持 `60-100 bpm` 之间的正常波动，而非显示 "0 bpm" 或 "NaN"。

    *   **测试目的**：防止物理感知注入的字段污染系统 Prompt（例如用户设置的模拟位置名字中带有注入攻击性词汇）。
    *   **预期结果**：`PromptAssembler` 应对地理位置、时间、心率等外部字符串进行转义或净化，确保其以标准的结构化文本呈现，LLM 不会发生语义偏离。
35. **用例 R3-T2-05：物理感知完全关闭时的彻底清除**
    *   **测试目的**：验证当用户在设置里全局关闭物理感知选项时，发送载荷的彻底净化。
    *   **预期结果**：发出的 Payload 绝对不含任何关于 Location、Heart Rate、Time 的 context 节点，最大程度保障隐私。

#### 【R4. 后台定时主动问候推送】
36. **用例 R4-T2-01：系统低电量/打盹模式 (Doze Mode) 下的任务调度**
    *   **测试目的**：验证在系统电池优化拦截下，WorkManager 对定时主动问候的保活或延后唤醒策略。
    *   **预期结果**：使用约束条件（如 `setRequiresBatteryNotLow(false)`）保证关键问候依然在合适的时间窗发出，或者降级为在退出 Doze 状态时立即补发。
37. **用例 R4-T2-02：后台静默网络超时与自动退避（Backoff）**
    *   **测试目的**：后台任务执行时，大模型接口连接超时或断网。
    *   **预期结果**：后台任务静默宣告失败，WorkManager 触发退避策略（如指数退避）在 15 分钟后重试，绝对不在通知栏弹窗提示用户网络出错。
38. **用例 R4-T2-03：连续多次触发时的通知合并逻辑**
    *   **测试目的**：如果前一天的问候通知未被用户点击，今天又生成了新的问候通知，测试通知栏的处理。
    *   **预期结果**：新生成的问候通知应更新旧的通知（或者合并为单一的伴侣关怀合集通知），不在通知栏堆叠刷屏。
39. **用例 R4-T2-04：后台生成问候时关联会话已被用户删除**
    *   **测试目的**：大模型生成后台问候时，其原本指定的 Active Session 被用户在前台删除了。
    *   **预期结果**：点击通知唤起 App 时，系统能够智能重建会话，或者将问候消息写入当前唯一的活跃会话，防止出现由于找不到 SessionId 而导致空指针异常。
40. **用例 R4-T2-05：AI 问候生成文本严重超长（大模型幻觉）**
    *   **测试目的**：大模型在后台没有严格遵循“50字以内”的指令，吐出了数千字的长文。
    *   **预期结果**：通知构建器使用 `NotificationCompat.BigTextStyle` 并在展示上限处做优雅的客户端硬截断（强制前 50 字加 "..."），在点击通知进入聊天后，则展示完整的大模型超长回复。

## 三、 测试实施技术建议与框架选择

为了使后续的 Implementer 代理能高效、无差错地实现上述测试，特提出以下具体的架构实施建议：

1.  **测试运行器选择**：
    由于所有界面均由 Jetpack Compose 构建，必须使用 **`createAndroidComposeRule<MainActivity>()`** 来创建测试规则。它可以接管 `MainActivity` 并允许测试脚本对 UI 树进行深度匹配和动作点击模拟。
2.  **异步数据与流式等待 (Idling Resources)**：
    大模型 SSE 接收以及 MCP JSON-RPC 调用都属于异步网络行为。测试框架中应当使用 `composeTestRule.waitUntil`，配置合理的超时时间（如 10 秒），以等待 `isThinking` 状态从 true 变为 false，从而确保在断言大模型回答前，打字机流式输出已经完全结束。
3.  **Mock 服务的搭建**：
    *   **LLM API Mock**：可以使用 `MockWebServer` 拦截大模型 API 的请求，并根据用例需求模拟返回包含特定 `tool_calls` 的 SSE 响应或文本。
    *   **MCP Mock 服务**：可在测试套件中内置一个简单的本地 HTTP/SSE 服务，在测试运行前启动，监听特定端口，用于模拟 MCP 管理器连接并提供工具列表及响应 JSON-RPC 结果。
4.  **硬件数据与 WorkManager 模拟**：
    *   **传感器 Mock**：将 `WatchDataRepository` 定义为依赖注入（DI）的单例接口。在测试环境中，替换为 `TestWatchDataRepository`，允许测试脚本动态更改心率数值。
    *   **WorkManager 测试**：使用 Android 官方的 `androidx.work:work-testing` 库。通过 `WorkManagerTestInitHelper` 初始化，并直接调用其测试 API 来立刻强制执行定时问候 Worker，而不需要让测试用例等待至真实深夜。
